============================================================
Confirmed Remaining Bugs
============================================================

[HIGH] Default credential/session secret fallback still present
- Files:
  - database.js (default ADMIN_USERNAME/ADMIN_PASSWORD fallback)
  - server.js (default SESSION_SECRET fallback)
- Risk:
  - If env vars are omitted in deployment, app can run with predictable secrets.
- Evidence:
  - Runtime warnings printed on startup confirming fallback mode.

[MEDIUM] Oversized JSON payload returns 500 instead of 413
- File: server.js (global error handler)
- Repro:
  - POST /api/expenses with body > 16kb
- Actual:
  - HTTP 500 with {"error":"Internal server error"}
  - Log shows PayloadTooLargeError: request entity too large
- Expected:
  - HTTP 413 Payload Too Large (clean client-facing response)
- Impact:
  - Incorrect error semantics; monitoring noise and avoidable 500s.

============================================================
Recommendations
============================================================

1) Fail fast for missing production secrets
- Remove insecure fallbacks for admin credentials and session secret in production mode.
- Exit process when required env vars are absent.

2) Handle body-parser limit error explicitly
- In error middleware:
  - if err.type === 'entity.too.large' -> return 413 with JSON error.

3) Optional hardening follow-up
- Add automated integration tests for:
  - auth lifecycle
  - CSRF rejection
  - rate-limiting threshold
  - 413 behavior on oversized payload
